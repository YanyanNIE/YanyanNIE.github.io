linux 线程

Posted 2019-06-14 | Modified: 2019-11-29 | In base | | views
: 3.5k | ≈ 13

简介

在传统的操作系统中, 拥有资源和独立调度的基本单位都是进程. 在引入线程的操作系统中, 线程是独立调度的基本单位, 进程是资源拥有的基本单位.

进程之间是相互独立的, 每个进程拥有自己的一块自身内存区域, 因此需要解决的问题是进程间的通信是; 多个线程之间是共享一块内存区域和一组资源的, 有可能相互影响, 因此会产生多个线程同时访问某一块数据的时候会导致数据不一致的问题, 需要解决的问题是线程之间的同步.

线程与同一个进程的其他线程共享进程所拥有的全部资源, 因此线程的切换要比进程切换的负担小.

同一个进程中的多线程之间可以并发执行, 一个线程可以创建和撤消另一个线程

thread-safe 概念

线程安全是多线程领域的问题, 可以简单理解为在多线程环境中, 多个线程同时访问相同的资源, 比如同一内存区时, 获得的结果在逻辑预期中, 那这样即是线程安全.

线程安全涉及多线程情况下共享资源的管理.

内核级线程与用户级线程

根据操作系统内核是否对线程可感知, 可以把线程分为内核线程和用户线程. 用户级线程由应用程序所支持的线程实现, 内核意识不到用户级线程的实现

内核线程是由操作系统维护的线程对象. 它是能够被处理器调度和执行的实际线程. 通常, 系统线程是具有权限设置, 优先级等的重量级对象. 内核线程调度器负责调度内核线程.

用户程序自己制作的“线程”不是内核线程. 每个用户线程都不能自己运行, 用户线程运行的唯一方法是: 告知某个内河线程要执行用户线程的相关代码. 用户线程必须与内核线程相关联, 因为用户线程本身只是用户程序中的一堆数据. 内核线程是系统中的真正线程, 所以为了让用户线程运行, 用户程序必须让其调度程序接受用户线程, 然后在内核线程上运行它.

用户线程和内核线程之间的映射可以是一对一映射, 即每个用户线程绑定一个内核线程; 也可以多个用户线程共享相同的内核线程(每次只运行一个用户线程), 或者也可以让单个用户线程在不同的内核线程(1:n)映射之间进行切换

linux 下使用线程

Linux 环境下创造线程需要声明的头文件:

pthread_create

作用: 创建线程函数

函数原型:

  • 返回值:

    • 成功: 0
    • 出错: 出错编号

  • 传入参数:

    • thread: 返回成功时, thread 指向的内存单元被设置为新创建线程的线程ID
    • attr: 用于制定线程属性
    • start_routine: 新创建的线程从 start_routine 函数的地址开始运行

    • arg: 向 start_routine 函数传入的参数结构.

      注意 start_routine 函数只有一个指针参数 arg, 如果需要向函数传递的参数不止一个, 那么需要把这些参数放到一个结构中, 然后把这个结构的地址作为arg的参数传入.

  • 错误代码:

    • EAGAIN: 资源不足, 无法创建另一个线程
    • EINVAL: attr 中的设置无效
    • EPERM: 无权设置 attr 中指定的调度策略和参数

pthread_attr_t

在 phread_create() 函数中, 第二个参数是线程属性设置, 一般设置为了NULL, 但是可以通过系统API进一步设置第二个参数, 细化创建的线程的属性.

  • scope (int) - 内核级线程/用户级线程: PTHREAD_SCOPE_SYSTEM, PTHREAD_SCOPE_PROCESS
  • stackaddr (void *)- 堆栈地址
  • stacksize ( size_t ) - 堆栈大小
  • guardsize: 栈保护区大小, 在线程栈顶留出一段空间, 防止栈溢出. 当栈指针进入这段保护区时, 系统会发出错误, 通常是发送信号给线程. 该属性默认值是PAGESIZE大小, 该属性被设置时, 系统会自动将该属性大小补齐为页大小的整数倍. 当改变栈地址属性时, 栈保护区大小通常清零.
  • detachstate (int) - 线程 join 或 detach 状态: 线程的结合或分离状态决定一个线程以什么样的方式来终止自己
    • PTHREAD_CREATE_JOINABLE: 可结合的线程能够被其他线程收回其资源和杀死; 在被其他线程回收之前, 它的存储器资源（如栈）是不释放的; 默认情况下线程是可结合的 (joinable)
    • PTHREAD_CREATE_DETACHED: 分离的线程是不能被其他线程回收或杀死的, 它的存储器资源在它终止时由系统自动释放
  • schedpolicy (int) - 线程的调度方式:
    • SCHED_OTHER(unix): 这个策略在“POSIX 标准 1003.4a”中作为定义的执行程序进行定义, 是默认的Linux分时调度（time-sharing scheduling）策略. 在每个时钟中断时重算运行线程的优先级值意味着该线程可能失去控制权, 因为它的优先级值已经超过了另一可分派线程的优先级值.
    • SCHED_FIFO: 先入先出调度策略. 这种策略的线程被调度后, 它会一直运行到结束, 除非被阻塞或有一个较高优先级的线程可分派, 它将自愿服从处理器的控制. 只有固定优先级的线程才能有 SCHED_FIFO 调度策略
    • SCHED_RR: 时间片轮转调度(Round-robin scheduling). 该策略是SCHED_FIFO基础上改进来的, 他给每个线程增加了一个时间片限制, 当时间片用完后, 系统将把该线程置于队列末尾. 放在队列尾保证了所有具有相同优先级的RR任务的调度公平. 只有固定优先级的线程才能有 SCHED_RR 的调度策略
  • schedparam (sched_param *) - 线程的调度参数
  • inheritsched (int) - 线程的继承性, 决定调度的参数是从创建的进程中继承还是使用在schedpolicy和schedparam属性中显式设置的调度信息: PTHREAD_INHERIT_SCHED, PTHREAD_EXPLICIT_SCHED

  • pthread_join 用于等待一个线程的结束, 也就是主线程中要是加了这段代码, 就会在加代码的位置卡主, 直到这个线程执行完毕才往下走.

  • pthread_exit 用于强制退出一个线程（非执行完毕退出）, 一般用于线程内部.

一般都是 pthread_exit 在线程内退出, 然后返回一个值. 这个时候就跳到主线程的 pthread_join 了（因为一直在等你结束）, 这个返回值会直接送到 pthread_join, 实现了主与分线程的通信.

注意 这个线程退出的返回值的格式是 void* , 无论是什么格式都要强转成 void 才能返回出来主线程（pthread_exit((void)tmp);）, 而这个时候 pthread_join 就去接这个值, 我们传进去一个 void 的地址也就是 `&（void）`

pthread 和 fork 混用的情况

来自于Unix下fork与pthread混用的问题

在POSIX标准中, fork 的行为是这样的: 复制整个用户空间的数据（通常使用copy-on-write的策略, 所以可以实现的速度很快）以及所有系统对象, 然后仅复制当前线程到子进程. 这里: 所有父进程中别的线程, 到了子进程中都是突然蒸发掉的.

其它线程的突然消失, 是一切问题的根源. 可能产生的最严重的问题是锁的问题.

因为为了性能, 大部分系统的锁是实现在用户空间的. 所以锁对象会因为 fork 复制到子进程中.

对于锁来说, 从OS看, 每个锁有一个所有者, 即最后一次lock它的线程.

假设这么一个环境, 在fork之前, 有一个子线程lock了某个锁, 获得了对锁的所有权. fork以后, 在子进程中, 所有的额外线程都人间蒸发了. 而锁却被正常复制了, 在子进程看来, 这个锁没有主人, 所以没有任何人可以对它解锁.

当子进程想lock这个锁时, 不再有任何手段可以解开了. 程序发生死锁.

线程-信号问题

在linux下, 每个进程都有自己的signal mask, 指定哪个信号被阻塞, 哪个不会被阻塞. 同时每个进程还有自己的signal action, 这个行为集合指定了信号该如何处理

  • 异常产生的信号: 只有产生异常的线程收到并处理: SIGPIPE、SIGEGV
  • pthread_kill产生的内部信号: 只有pthread_kill参数中指定的目标线程收到并处理
  • 外部使用kill命令产生的信号: 遍历所有线程, 直到找到一个不阻塞该信号的线程; 一般从主线程找起, 且只有一个线程能收到: SIGINT、SIGHUP

每个线程都有自己独立的signal mask, 但所有线程共享进程的signal action

线程临界区

POSIX将互斥锁定义为 pthread_mutex_t 类型变量, 它具有一组属性, 类型为 pthread_mutexattr_t

  • Mutex 是一个二进制信号量. 它有两种可能的状态: 自由或锁定
    • 在自由情况下, 只有一个Pthread可以获得让临界区进入锁定状态
    • 临界区已锁定的状态下, 任何请求都将被阻止
